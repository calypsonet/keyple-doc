1. Commands Extensions - 'Calypso processing' API (Low layer)

1.1. generic Calypso Command Builder & Response Parser

PO & CSM APDU request builder classes have to extend ApduCommandBuilder:

  • the daughter sub-classes have to provide the dedicated constructor to build the specific APDU,

  • the mother class defines the ApduRequest attribute.

PO & CMS APDU response parser classes have to extend ApduResponseParser:

  • the daughter sub-classes have to provide the dedicated getters to access to the specific fields of the APDU response,

  • the mother class defines the ApduResponse attribute, and the generic abstract constructor.

Generic classes and interfaces necessary to manage APDU Command Builders & Response Parser are contained in the ‘cna.sdk.commands’ package.

Calypso Commandss Sets are defined in the ‘cna.sdk.commands.calypso’ sub-package:

  • Generic parameters for Calypso PO & CMS commands’ builders & responses’, and their associations are registered in the CalypsoCommands Enumeration.

  • Portable Object’s command’s builders are in the ‘org.keyple.calypso.commands.po.builder’ sub-package,

  • Portable Object’s response’s parsers are in the ‘org.keyple.calypso.commands.po.parser’ sub-package,

  • Secure Module’s command’s builders are in the ‘org.keyple.calypso.commands.csm.builder’ sub-package,

  • and Secure Module’s command’s parsers are in the ‘org.keyple.calypso.commands.csm.parser’ sub-package.

An application could manage additional Commands by providing another implementation of the CommandsTable Interface.

1.2.4. abstract Class: PoCommandBuilder

This abstract class extends ApduCommandBuilder, it has to be extended by all PO command builder classes, it manages the current default revision for PO commands.

  • static Attribute: PoRevision defaultRevision

    • the current default revision for PoCommandBuilder.

  • abstract Method: PoCommandBuilder(CalypsoCommands reference, ApduRequest request)

    • the generic abstract constructor to build an PO request with a Calypso command’s reference and a byte array.

1.2.5. Enumeration: CsmRevision

This enumeration registers the different mode of CSM. Currently the specific CSM feature identified is the support of ISO 7816-4 commands’ (C1 & S1E SAM with commands in CLA 80h), or proprietary commands (S1D SAM with commands in CLA 94h).

  • Literals:

C1

S1D

1.2.6. abstract Class: CsmCommandBuilder

This abstract class extends ApduCommandBuilder, it has to be extended by all CSM command builder classes, it manages the current default revision for PO commands.

  • static Attribute: CsmRevision defaultRevision

    • the current default revision for CsmCommandBuilder.

  • abstract Method: CsmCommandBuilder(CalypsoCommands reference, ApduRequest request)

    • the generic abstract constructor to build an CSM request with a Calypso command’s reference and a byte array.

1.2.7. Class: InconsistentCommandException

Extends the Exception class of the environment.

1.2.8. Interface (or abstract Class): SendableInSession

This object could be defined as an interface in Java, or as an abstract class in C++.

This interface (or abstract class) has to be implemented (extended) by the builders of all PO commands, which can be sent inside a secure session.

1.3. Calypso Portable Object command’s set (Middle Layer)

All the related elements are located in the package 'org.keyple.calypso.commands.po'.

  • Builders of PO command are included in the sub-package 'org.keyple.calypso.commands.po.builder'. Related classes extend AsbstractPoCommandBuilder.

  • Parsers of PO response are included in the sub-package 'org.keyple.calypso.commands.po.parser'. Related classes extend AbstractApduResponseParser.

//// All these elements are located in the package 'org.keyple.calypso.commands.po.builder', all included classes extend PoCommandBuilder.

1.3.1. Class: GetDataFciCmdBuild

This class extends PoCommandBuilder and implements SendableInSession, it provides the dedicated constructor to build the Get Data for ‘FCI’ APDU command.

  • Method: GetDataFCICmdBuild (PoRevision[0..1] revision)

    • The CalypsoCommands reference is defined with the value ‘PO_GET_DATA_FCI’.

    • If the optional revision parameter is missing, use the current PoCommandBuilder.defaultRevision. In revision 1 & 2, the class byte is set at 94h (otherwise at 00h).

  • Method: GetDataFCICmdBuild (ApduRequest request) throw InconsistentCommandException

    • The CalypsoCommands reference is defined with the value ‘PO_GET_DATA_FCI’.

    • Bytes low level constructor: throw an exception if the request’s instruction code isn’t the one defined for ‘PO_GET_DATA_FCI’ CalypsoCommands’ reference, otherwise call the super constructor.

Remark:

  • In Java or C++, several constructors could be defined to manage the optional parameters.

  • About the different modes of the Calypso PO Get Data command:

    • Through a CommandsTable, an ApduCommandBuilder could be linked to only one ApduResponseParser.

    • Depending on P1/P2 parameters the Calypso PO GetData command could returned different kind of structured response. It seems preferable to defined a specific ApduResponseParser for each mode of the Get Data command. Then it makes sense to define a specifc ApduCommandBuilder for each mode of the Get Data command.

To study,

  • Maybe an abstract ‘GetDataCmdBuild’ class could be defined to support all the modes of the Get Data command, and then GetDataFCICmdBuild could extend it. In this case the constructor of GetDataCmdBuild may contain a parameter specifying the mode of the command, all modes being defined in an enumeration.

    • The CalypsoCommands reference is defined with the value ‘PO_OPEN_SESSION’.

    • If the optional revision parameter is missing, use the current PoCommandBuilder.defaultRevision. In revision 1 & 2, the class byte is set at 94h (otherwise at 00h). In revision 2, the presence of the KVC is asked.

  • Method: OpenSessionCmdBuild (ApduRequest request) throw InconsistentCommandException

    • Bytes low level constructor: throw an exception if the request’s instruction code isn’t the one defined for ‘PO_OPEN_SESSION’ CalypsoCommands’ reference, otherwise call the super constructor.

    • The CalypsoCommands reference is defined with the value ‘PO_CLOSE_SESSION’.

    • If the optional revision parameter is missing, use the current PoCommandBuilder.defaultRevision. In revision 1 & 2, the class byte is set at 94h (otherwise at 00h).

    • The optional parameter terminalSessionSignature could contain 4 or 8 bytes.

  • Method: CloseSessionCmdBuild (ApduRequest request) throw InconsistentCommandException

1.7. Calypso High layer – Calyspo Secure Transaction API

The high level Calypso transaction module is defined on top of the PO & CSM commands’ API, its classes and interface will be contained in the package ‘cna.sdk.calypso.transaction’,

The aim of this transaction module is to automate security operations linked to a session or a transaction. Through this layer of API, an application doesn’t require to manage directly CSM operations (like proposed by lots of contactless readers which embed CSM slots.

1.7.1. Class: PoPlainSecureSession

A non-encrypted secure session with a Calypso PO requires the management of two ProxyReader in order to communicate with both a Calypso PO and a CSM.

  • private Attribute: ProxyReader poReader;

  • private Attribute: ProxyReader csmReader;

    • The CSM used to manage the session with the PO

  • private Attribute Boolean poIdentified; (false by default)

  • private Attribute Boolean sessionOpened; (false by default)

  • private Attribute Boolean encryptionSupported; (false by default)

  • private Attribute Boolean encryptionActivated; (false by default)

  • Method: PoSecureSession(ProxyReader poReader, ProxyReader csmSessionReader, ByteArray csmSettings)

    • csmSettings not defined: default AID, default session keyset…

  • Method: SeResponse processIdentification (ByteArray poAid, SendableInSession[0..*] poCommands_OutsideSession)

    • On poReader, generate a SeRequest for the specified PO AID, with keepChannelOpen set at true, and apduRequests defined with the optional poCommands_OutsideSession.

    • Returns the corresponding SeResponse.

    • Identifies the serial & the revision of the PO from FCI data.

    • On csmSessionReader, automatically operate the Select Diversifier & the Get Challenge.

  • Method: SeResponse processOpening (OpenSessionCmdBuild openCommand, SendableInSession[0..*] poCommands_InsideSession)

    • On poReader, generate a SeRequest with the current selected AID, with keepChannelOpen set at true, and apduRequests defined with openCommand and the optional poCommands_InsideSession.

    • Returns the corresponding SeResponse (for openCommand and poCommands_InsideSession).

    • Identifies the session PO keyset.

    • On csmSessionReader, automatically operate the Digest Init & potentially several Digest Update Multiple.

  • Method: SeResponse processProceeding (SendableInSession[1..*] poCommands_InsideSession)

    • On poReader, generate a SeRequest with the current selected AID, with keepChannelOpen set at true, and apduRequests defined with the poCommands_InsideSession.

    • Returns the corresponding SeResponse (for poCommands_InsideSession).

    • On csmSessionReader, automatically operate potentially several Digest Update Multiple.

  • Method: \{SeResponse + Boolean} processClosing (SendableInSession[0..*] poPredetermineCommands_InsideSession, CloseSessionCmdBuild closeCommand, PoGetChallengeCmdBuild ratificationCommand)

    • On csmSessionReader, automatically operate potentially several Digest Update Multiple, and the Digest Close

    • Identifies the terminal signature.

    • On poReader, generate a SERequest with the current selected AID, with keepChannelOpen set at false, and apduRequests defined with poCommands_InsideSession, closeCommand, & ratificationCommand.

    • Identifies the PO signature.

    • On csmSessionReader, automatically operates the Digest Authenticate.

    • Returns the corresponding SeResponse and the Boolean status of the authentication.
